# Error Catalog

Complete catalog of every error message the user can encounter in BoydCode.
Each entry documents the exact text, trigger condition, severity, recovery
action, and implementation location.

All file paths are relative to the repository root.

---

## Exit Codes

Defined in `src/BoydCode.Domain/Enums/ExitCode.cs`:

| Code | Name | Meaning |
|---|---|---|
| 0 | `Success` | Normal exit |
| 1 | `GeneralError` | Unrecoverable error or unhandled exception |
| 2 | `AuthenticationError` | Login/OAuth failure |
| 3 | `ProviderError` | LLM provider communication failure |
| 4 | `ConfigurationError` | Missing or invalid configuration |
| 5 | `NetworkError` | Network connectivity issue |
| 130 | `UserCancelled` | User pressed Ctrl+C during session |

## 2. Provider Errors

These errors are generated by `AgentOrchestrator.FormatProviderError()` and
`ClassifyAndSuggest()` in `src/BoydCode.Application/Services/AgentOrchestrator.cs`.
They appear when the LLM API call fails during an agent turn.

| ID | Message Pattern | Trigger | Severity | Recovery | Stream | Implementation |
|---|---|---|---|---|---|---|
| ERR-PROV-01 | `{extracted message}` + `Suggestion: Check your API key with /provider setup or pass --api-key.` | Exception message contains "API KEY", "UNAUTHORIZED", "AUTHENTICATION", "PERMISSION_DENIED", or "FORBIDDEN" | Recoverable | Run `/provider setup` or pass `--api-key` | stderr | `AgentOrchestrator.cs:462-467` |
| ERR-PROV-02 | `{extracted message}` + `Suggestion: Wait a moment and retry, or switch providers with /provider setup.` | Exception message contains "RATE LIMIT", "TOO MANY REQUESTS", "429", or "QUOTA" | Recoverable | Wait and retry; or switch provider | stderr | `AgentOrchestrator.cs:469-473` |
| ERR-PROV-03 | `{extracted message}` + `Suggestion: Start a new session or switch to a model with a larger context window.` | Exception message contains "CONTEXT", "TOKEN LIMIT", "TOO LONG", or "MAX.*TOKEN" | Recoverable | Use `/context compact` or start new session | stderr | `AgentOrchestrator.cs:475-479` |
| ERR-PROV-04 | `{extracted message}` + `Suggestion: Check your internet connection and try again.` | Exception is `HttpRequestException` or message contains "CONNECTION", "TIMEOUT", "NETWORK" | Recoverable | Check network connectivity | stderr | `AgentOrchestrator.cs:481-485` |
| ERR-PROV-05 | `{extracted message}` + `Suggestion: The provider may be experiencing issues. Try again in a few moments.` | Exception message contains "500", "SERVER ERROR", "OVERLOADED", "503" | Recoverable | Wait and retry | stderr | `AgentOrchestrator.cs:487-491` |
| ERR-PROV-06 | `{extracted message}` (no suggestion) | None of the above patterns match | Recoverable | Review the error message; check provider status | stderr | `AgentOrchestrator.cs:493` |
| ERR-PROV-07 | `No LLM provider configured. Use /provider setup to configure one.` | `RunAgentTurnAsync` called when `_activeProvider.IsConfigured` is false | Recoverable | Run `/provider setup` | stderr | `AgentOrchestrator.cs:137` |
| ERR-PROV-08 | `Reached maximum tool call rounds ({maxToolRounds}). Stopping to prevent runaway execution.` | Agent loop hits 50 rounds without `end_turn` stop reason | Recoverable | Start new message or use `/context compact` | stderr | `AgentOrchestrator.cs:217` |

**Error extraction:** `FormatProviderError()` attempts to extract a
human-readable message from JSON error bodies embedded in exception messages
(common with Anthropic, OpenAI, and Google SDKs). It checks for
`{ "error": { "message": "..." } }` (nested) and `{ "message": "..." }`
(flat) patterns. If no JSON is found, it falls back to the inner exception
message if different from the outer.

## 13. Consistency Analysis

### 13.1 Errors on stdout Instead of stderr

The following errors use `SpectreHelpers.Error()` or raw
`AnsiConsole.MarkupLine("[red]Error:[/] ...")`, which write to **stdout**.
Only `SpectreUserInterface.RenderError()` writes to stderr.

**Affected locations:**

- All `SpectreHelpers.Error()` calls: `ConversationsSlashCommand.cs:39`,
  `ContextSlashCommand.cs:93,403,445,494,512`, `ContextSlashCommand.cs:79,94`,
  `LoginCommand.cs:35`, `ProviderSlashCommand.cs:143`,
  `ProjectSlashCommand.cs:372`
- All raw `AnsiConsole.MarkupLine("[red]Error:[/] ...")` calls in slash
  commands: `ProjectSlashCommand.cs:108,230,383,489,496`,
  `JeaSlashCommand.cs:186,220,296,450,457,534,563,570,596,603,626,655,662,668`,
  `ConversationsSlashCommand.cs:127,190,197`, `ProviderSlashCommand.cs:177,231`
- All `AnsiConsole.MarkupLine("[red]...[/]")` calls in `LoginCommand.cs`:
  lines 42, 52, 87, 92, 101, 245, 246

**Impact:** These errors are not visible when stdout is piped/redirected,
and they are mixed into the normal output stream. Only
`AgentOrchestrator`-originated errors (via `_ui.RenderError()`) correctly
go to stderr.

### 13.2 Errors Missing the Three-Part Pattern

The recommended error pattern is: (1) what happened, (2) why, (3) what to
do about it. The following errors lack recovery guidance:

| Location | Message | Missing |
|---|---|---|
| `LoginCommand.cs:42` | `Provider '{Provider}' does not support OAuth login.` | No alternative action suggested |
| `LoginCommand.cs:52` | `OAuth client ID is required. Please try again.` | No guidance on where to find it |
| `LoginCommand.cs:87` | `Login timed out. Please try again.` | No explanation of why it timed out |
| `LoginCommand.cs:92` | `{ex.Message}` (raw exception) | No context or recovery guidance |
| `LoginCommand.cs:101` | `Failed to exchange authorization code for tokens.` | No explanation of common causes |
| `ChatCommand.cs:100` | `Warning: Directory does not exist: {path}` | Uses `RenderError` for a warning; misleading severity |

### 13.3 Errors Using Raw Markup Instead of SpectreHelpers

`LoginCommand.cs` does not use `SpectreHelpers` at all for its error/success
messages. Instead, it uses raw `AnsiConsole.MarkupLine("[red]...[/]")` and
`AnsiConsole.MarkupLine("[green]...[/]")`. This creates a different visual
language compared to the rest of the application:

- No `Error:` prefix on error messages (lines 42, 52, 87, 92, 101)
- `[green]Successfully logged in![/]` instead of `[green]v[/] ...` (line 114)
- `[yellow]...[/]` instead of `[yellow]Warning:[/] ...` (line 134)

### 13.4 Warning Rendered via RenderError

`ChatCommand.cs:100` renders `Warning: Directory does not exist: {path}`
using `_ui.RenderError()`. This writes to stderr with the bold red
`Error:` prefix, but the message text starts with "Warning:", creating
a confusing `[red bold]Error:[/] [red]Warning: Directory does not exist...[/]`
output.

### 13.5 Inconsistent Entity Name Bolding

Some error messages bold the entity name, others do not:

- **Bolded:** `Project [bold]{name}[/] not found.` (ProjectSlashCommand)
- **Not bolded:** `No active session.` (ConversationsSlashCommand, ContextSlashCommand)
- **Not bolded:** `No LLM provider configured.` (AgentOrchestrator)

The pattern is consistent within each file but inconsistent across files.

### 13.6 Auto-Save Failures Are Silent

`AgentOrchestrator.AutoSaveSessionAsync()` catches all exceptions and only
logs them. The user is never informed that their session failed to save.
This is intentional (auto-save is best-effort) but could lead to data loss
without the user knowing.
